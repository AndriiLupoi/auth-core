## 1. Призначення системи

AuthCore є централізованим мікросервісом автентифікації та авторизації, що забезпечує єдиний захищений механізм входу для всіх клієнтських застосунків організації. Сервіс виступає **єдиним джерелом достовірності** щодо ідентичності користувача (Single Source of Truth).

Формат взаємодії — виключно REST API із підтримкою JSON. Система не має власного UI для кінцевих користувачів — вся взаємодія відбувається через API або через UI клієнтських застосунків.

## 3. Межі системи

### У сфері відповідальності системи

- Управління обліковими записами користувачів (реєстрація, верифікація, відновлення)
- Видача та валідація JWT access-токенів та refresh-токенів
- Підтримка стандарту OAuth 2.0 та OpenID Connect
- Управління ролями та правами доступу (RBAC)
- Ведення журналу подій безпеки (audit log)

### Поза межами системи (out of scope)

- Бізнес-логіка клієнтських застосунків
- Авторизація на рівні ресурсів клієнтів
- Зберігання персональних даних користувача (ім'я, адреса тощо)

## 5. Нефункціональні вимоги

### Продуктивність

- Система повинна обробляти не менше **1000 запитів на хвилину**
- Середній час відповіді на запит автентифікації — не більше **200 мс**
- Час валідації JWT-токена — не більше **10 мс**

### Доступність

- Доступність системи — не менше **99.9%** (не більше 8.7 год простою на рік)
- Відновлення після збою — не більше **60 секунд** (автоматичний рестарт контейнерів)

### Безпека

- Усі паролі зберігаються у вигляді хешу bcrypt (cost factor ≥ 12)
- JWT-токени підписуються алгоритмом **RS256**
- Access-токен має TTL **15 хвилин**, refresh-токен — **30 днів**
- Усі з'єднання виключно по **HTTPS/TLS 1.2+**
- Захист від brute-force: блокування після **5 невдалих спроб** входу

### Масштабованість

- Горизонтальне масштабування до **10 екземплярів** сервісу без змін у коді
- Stateless-архітектура — стан не зберігається в пам'яті процесу

## 6. Обмеження та припущення

- Система розгортається виключно у контейнеризованому середовищі (Docker / Kubernetes)
- Клієнти зобов'язані передавати токен у заголовку `Authorization: Bearer <token>`
- Email-провайдер підключається як зовнішній сервіс (SMTP / SendGrid)
- SMS-провайдер (Twilio або інший) підключається через окремий адаптер

---

## 7. Основні функціональні сценарії

1. Реєстрація нового користувача з верифікацією електронної пошти
2. Вхід у систему з видачею пари JWT-токенів (access + refresh)
3. Оновлення access-токена через refresh-токен
4. Двофакторна автентифікація (2FA) через TOTP-додаток або SMS
5. Управління сесіями: перегляд активних сесій та примусовий вихід
6. Федеративна автентифікація через OAuth 2.0 провайдерів (Google, GitHub)
7. Скидання пароля через захищене посилання на email
8. Адміністративне управління користувачами та ролями
